# Scout 2: 思路目录 Doctrine 深扫报告

结论先行：`_原料/思路/` 这 3 个文件不是普通灵感碎片，而是**用户真实在用的创作操作系统上层宪法**。其中，`待整理思路 2` 是面向玄幻黑暗爽文的生产级 skill，`待整理思路 3` 是面向长篇仓库协作的文件化规划 skill，`待整理思路1` 则像它们共同的母规则池：搜索纪律、事实纪律、去 AI 味、逻辑死磕、开篇钩子、人物不崩、爽点兑现、资料落地、包装转化，全都在这里。蒸馏时不能把它们当“素材”重做；更准确的做法是：**Meta 层提炼宪法，Platform 层保留 skill 形态，Foundation/RAG 只做补强，不碰既有意图与工作流主干。**

## 思路 1 - 10 条 SOP

这里的“10 条 SOP”表面上是搜索/答题/排版规范，实质上已经延伸成一整套“创作 + 研究 + 审校”母规则。按蒸馏可注入性，建议拆成“必须做 / 必须不做”。

### 必须做

1. **不确定即搜索**  
   只要事实、数据、时效性、术语、政策、年代、真实世界细节存在不确定，就要联网核验。

2. **搜索必须迭代**  
   首轮查不到不算完成，必须改关键词、换中英文、换专业术语、做反向求证，直到锁定信息。

3. **关键信息交叉验证**  
   严禁单一信源拍板；核心事实至少 2-3 个权威来源交叉核对，冲突要显式指出。

4. **答案必须综合而非搬运**  
   搜索结果只能消化后重组，不能原样罗列；输出要“就事论事”，贴合用户真正意图。

5. **默认中文、真人口吻、去机械感**  
   语言要自然、直接、有力度，允许轻微幽默或俏皮，但不能变成表演型贫嘴。

6. **回答必须带原因与机制**  
   给步骤、结论、建议时，要顺带解释背后的原理、约束或判断依据。

7. **上下文绑定回答**  
   禁止孤立作答；必须把当前问题放进前文设定、已有文本、用户目标里一起判断。

8. **格式服务阅读与判断**  
   合理用 Markdown 层级、表格；LaTeX 只给数学；复杂结构在 token 允许时再图示，不为炫技而排版。

9. **输出前自校对**  
   核查事实、术语、符号、数值、逻辑链是否一致，再交付。

10. **所有内容必须对用户实际事业有用**  
   不是陪聊，不是花活；答案要能推动作品、决策、创作质量或协作效率。

### 必须不做

1. **不能假装知道**：不确定还不查，是这里最核心的禁忌之一。  
2. **不能只靠单源**：尤其历史、金融、军事、政策、地理、年代细节。  
3. **不能机械复述搜索结果**：文件反复强调“禁止搬运”。  
4. **不能输出 AI 腔**：包括模板句、空心高级词、无对象的泛泛抒情。  
5. **不能脱离上下文硬答**：尤其人物、设定、章节、资料类任务。  
6. **不能为了完整而堆砌背景**：信息要分层植入，不是一股脑灌给读者。  
7. **不能让人物降智推进剧情**：要反复追问“为什么这么做、是否符合利益、是否符合人设”。  
8. **不能把设定写成空架子**：世界观、金手指、资源、情绪都要落到事件。  
9. **不能让配角只剩工具功能**：配角必须有算盘、反扑、恐惧、利益链。  
10. **不能让包装与正文脱节**：书名、简介、开篇钩子必须和主卖点同频。

### 思路 1 的真正价值

它不是单纯的“10 条答题规则”。它已经扩展成 4 组上层母规范：

- **事实工作流母规范**：搜索、交叉验证、解释机制、格式纪律。
- **创作母规范**：show don't tell、冲突前置、欲望驱动、钩子、反馈。
- **人物母规范**：人设标签、反差细节、配角 B 面、关系要事件驱动。
- **避雷母规范**：16 组常见雷点，覆盖开篇、节奏、世界观、配角、主线、爽点、金手指、包装、排版、情感推进等。

换句话说，思路 1 是思路 2/3 的“上位法”。

## 思路 2 - dark-fantasy-ultimate-engine 画像

### 基本身份

- **名称**：`dark-fantasy-ultimate-engine`
- **定位**：为“自己的玄幻小说”执行正文续写、改写、扩写、卷纲整理、设定整理、质量审查、账本维护。
- **任务域**：正文创作 / 设定工作 / 质量审查 / 账本维护 四类。
- **工作语言**：简体中文。

### 目标语气与总风格

这个 skill 的目标不是一般玄幻，而是非常明确的一个窄通道：

- 高压迫感
- 高反馈
- 强因果
- 凶性
- 掠夺感
- 黑暗宇宙质感
- 无女主
- 杀伐果断
- 暴力美学

它的风格控制非常强：**所有创作性输出必须锁定玄幻小说风格**，而且显式禁止混入都市、科幻、游戏文、轻小说、影视剧本、悬疑刑侦、历史演义、散文议论、玩梗吐槽等口吻。这不是建议，是硬锁定。

### 任务判定逻辑

这是一个很成熟的“先分流、后执行” skill：

1. **正文创作**：续写、补写、改写、扩写章节。  
2. **设定工作**：世界观、人物卡、卷纲、势力、战力逻辑。  
3. **质量审查**：雷点、OOC、节奏、爽点、冲突闭环。  
4. **账本维护**：状态卡、微粒进度、资源折算、待回收伏笔、章节结算。  
5. **轻量讨论豁免**：如果只是创意聊天，不要强行套章节生产流程。

这个“不要过度启动流程”的判断很重要，说明它不是死板流水线，而是按任务幅度切换档位。

### 信息源优先级

这是思路 2 最成熟的一部分之一，已经具备明确的真值层级：

1. 用户本轮明确要求与硬约束  
2. 用户本轮贴出的正文 / 片段 / 设定补丁 / 修改指令  
3. `99_当前状态卡.md`  
4. 最近已存在正文章节  
5. 设定总索引  
6. 设定圣经中对应主定义文件  
7. 本地拆解知识库

并且给了冲突解决法：

- 最新正文高于旧大纲
- 当前时间点判断优先看状态卡
- 用户明确要求吃书修订时，可覆盖旧设定，但要说明覆盖了什么
- 拆解知识库只能借节奏与结构，不能搬桥段、台词、人物模板

这本质上是一个**写作领域的 source-of-truth 治理模型**。

### 核心设定关键词

从全文高频和硬约束来看，思路 2 锁定的世界/作品关键词包括：

- 四重天堑
- 内宇宙四阶
- 无女主
- 黑暗宇宙
- 极端掠夺
- 微粒进度 / 微粒账本
- 资源折算
- 待回收伏笔池
- 当前状态卡
- 章节结算
- 四重文档协同：正文 / 状态卡 / 资源账本 / 伏笔池

其中“微粒开启数 840000000 硬锚点”尤其关键：这说明它不是纯文风 skill，而是**带数值真值域的长篇生产系统**。

### 上下文加载哲学

这个 skill 非常反感“每次读满 100 章”的偷懒式保险策略，它强调：

- 先读状态卡
- 再按任务决定是否补读最近 3-10 章
- 只有长线伏笔、连续大战、跨地图推进明显相关时才扩窗
- 设定工作就读设定总索引和主定义文件，不要机械追正文

这和思路 3 的增量热更新哲学高度同构：**因果闭环 > 机械堆量**。

### 工作流骨架

正文任务的 5 步几乎已经是可执行 SOP：

1. 建立实时上下文  
2. 校验动机、信息边界与收益  
3. 识别章节类型（战斗 / 布局 / 过渡 / 回收）  
4. 施加场景压力  
5. 做外显一致性检查，再输出正文 / 修改结果 / 结算

这套流程的成熟点在于：它把“写作前自检”外显成表，而不是隐藏在黑箱思维里。

### 账本与状态系统

思路 2 不是单一写作 prompt，而是自带四套运行时记录系统：

- `99_当前状态卡.md`：当前时间点唯一覆盖文件
- `particle_ledger.md`：微粒与资源账本
- `pending_hooks.md`：待回收伏笔池
- 正文前后的 `【写作自检】` / `【章节结算】`

它还规定：

- 只改正文不改状态卡，不合格
- 只写收益不落微粒增量，不合格
- 完成大剧情节点后不更新伏笔池，不合格
- 长篇维护优先用 Markdown 表格，不要散文式自由文本

这说明它的本体是：**带账本的小说生产引擎**。

### 锁定风格 / 禁止风格

**锁定：**

- 玄幻小说质感
- 冷酷、护短、利己、斩草除根
- 暴力美学
- 黑暗宇宙
- 利益驱动
- 强因果反馈
- 场景化落地而非百科讲设定

**禁止：**

- 都市腔
- 科幻腔
- 游戏系统播报腔
- 轻小说吐槽腔
- 影视剧本腔
- 历史演义腔
- 散文抒情腔
- 暧昧救赎 / 后宫 / 圣母式留手
- 用大段设定说明替代事件推进
- 用“暴涨/海量/难以估量”跳过数值结算

### 好 / 雷判定标准

**好稿标准**：

- 当前锚点清楚
- 敌我已知信息边界清楚
- 收益可量化
- 伏笔状态可追溯
- 每场戏至少推进一项：信息、地位、资源、伤亡、仇恨、境界
- 三章内有明确反馈
- 配角 / 反派有脑子、有误判、有反扑

**雷稿标准**：

- OOC
- 设定冲突
- 敌方信息越界
- 战力崩坏
- 微粒数值跳变
- 伏笔失管
- 节奏拖沓
- 语言机械
- 词汇疲劳
- 知识库污染正文

### 这里的“sub-skill”实际画像

文件没有显式拆成多个 skill 文件，但内部已经形成若干可分发的功能子模组：

1. 正文创作引擎  
2. 设定整理器  
3. 质量审查器  
4. 状态卡维护器  
5. 微粒账本维护器  
6. 伏笔池维护器  
7. 章节结算器  
8. 冲突 / 收益 / 因果自检器

也就是说，它虽然名义上是一个 skill，结构上已经接近一个小型平台。

## 思路 3 - planning-with-files 画像

### 基本身份

- **名称**：`planning-with-files`
- **版本号**：`9.2.0-repo-grounded`
- **定位**：面向“当前仓库中的小说项目”的文件化规划与长篇一致性维护 skill。
- **关键口号**：把上下文当易失内存，把文件系统当持久记忆。

### 核心原则

这个 skill 的本体不是文风引擎，而是**仓库型小说工作流引擎**。它处理的是：

- 正文续写
- 设定整理
- 历史考据
- 质量审查
- 跨会话接力
- 规划文件维护

它明确反对 4 件事：

- 不伪造已读内容
- 不机械追读 100 章
- 不滥用联网
- 不暴露思维链

### 真实仓库适配能力

思路 3 非常“repo-grounded”，不是抽象 workflow：

- 直接点名根目录分卷正文 `卷1_001-020.md` 到 `卷13_241-260.md`
- 注意 `卷10_181-200 .md` 文件名里多了一个空格
- 点名 `character_relationships.md`
- 点名 `task_plan.md` / `findings.md` / `progress.md`
- 点名 `小说资料/`、`小说资料2/`、`imported_history/`
- 点名 `57小说/` 仅作对标语料库

这说明它不是泛用 prompt，而是已经与真实项目工作区深度耦合的生产技能。

### 任务判定逻辑

它把任务分成 5 类：

1. 正文创作  
2. 设定工作  
3. 历史考据  
4. 质量审查  
5. 规划维护

并给出豁免条件：轻量提问、单段润色、局部讨论，不强开全套规划文件。

这跟思路 2 一样，强调**按任务复杂度启动不同层级流程**。

### 信息源优先级

它的真值优先级比思路 2 更偏“仓库协作”：

1. 用户本轮明确要求与硬约束  
2. 用户本轮贴出的正文 / 设定 / 修改说明  
3. 最近相关正文或资料文件  
4. `task_plan.md` / `findings.md` / `progress.md`  
5. 本地参考资料与导入历史  
6. 联网核验结果  
7. 通用经验与类比

这里最关键的一句是：**联网结果只能补真实世界事实，不能覆盖小说中已写明的剧情事实。**

### 文件化工作流原则

思路 3 最大的价值，是把复杂创作任务拆进三份持久化文件：

- `task_plan.md`：目标、阶段、关键问题、决策、错误
- `findings.md`：事实、资料结论、人物状态、设定补丁、未决问题
- `progress.md`：本会话进度、动作、验证结果、阻塞项

使用规则也很成熟：

- 仓库里已存在则先回读、复用、补写，不要平行新建
- 有新发现、新约束就写 `findings.md`
- 每完成阶段或验证就写 `progress.md`
- 正文 / 卷进度推进后要同步更新计划与进度，防止正文与记录脱节
- 失败不能沉默重试，要记错误与新方案

这本质上是**把 AI 的短期上下文外置为文件化记忆层**。

### 渐进式上下文加载

思路 3 明确给了两种模式：

#### 模式 A：全量冷启动

适用于：

- 全书梳理
- 伏笔复盘
- 重做卷纲
- 全书级审查
- 设定重构

动作：列文件、建全局拓扑、抽人物状态 / 关键事件 / 时间线 / 未回收伏笔、落 `findings.md`。

#### 模式 B：增量热更新（默认）

适用于日常续写、润色、修文、局部审查。

动作：

1. 定位最新章节与直接相关文件  
2. 默认读最近 `3-10` 个相关章节或资料文件  
3. 长线伏笔 / 跨卷人物 / 重大伤势 / 长期交易链才扩到 `10-30`  
4. 只有明确要求全链复盘才继续扩窗

**硬约束**非常重要：

- 不能把“读得越多越安全”当默认
- 不能声称读过 100 章，除非真的读了
- 续写前必须确认最近相关章节的人物状态、公开情报、资源、伏笔
- 一旦发现问题依赖更早章节，要立刻扩窗

这套哲学跟思路 2 完全同宗：反对假勤奋，强调因果闭环。

### 目录约定与路径纪律

思路 3 指明了真实文件结构：

- `character_relationships.md`
- `task_plan.md`
- `findings.md`
- `progress.md`
- `小说资料/`
- `小说资料2/`
- `imported_history/`
- `57小说/`

同时明确：

- 用户给路径时优先用用户路径
- 缺目录要直说
- 不要把别的机器上的绝对路径写成唯一真路径
- `57小说/` 只能做节奏比较、对标、避雷、拆解，不是本书正史，不得直接挪桥段

这是一种很强的**路径真实主义**。

### allowed-tools 与 hooks

显式 allowed-tools：

- `read_file`
- `write_file`
- `replace`
- `glob`
- `web_fetch`
- `google_web_search`

显式 hook：

- 匹配 `write_file|replace|read_file`
- 执行一条提示命令：`【planning-with-files】正在同步计划文件与上下文锚点...`

这意味着思路 3 已经不只是文案约束，而是**带工具白名单与执行期提示钩子的半运行时 skill**。

### 隐式自检规则

虽然它没有把“隐式自检”单独命名成模块，但实际存在一整套默认检查：

- 当前时间 / 地点 / 对手 / 即时收益目标是否锁定
- 规划文件是否需要先读 / 先初始化
- 沈轩此刻利益最大化选择是什么
- 这场冲突为何非做不可
- 配角是否有目标 / 恐惧 / 筹码 / 误判
- 年代细节 / 地点习惯 / 身份权限是否兼容现实与既有正文
- 每段是否带来新信息 / 态度变化 / 利益变化
- 审查输出必须按“问题 -> 证据 -> 最小修法”

这说明它不是“写个计划文件就完事”，而是把计划文件和正文逻辑严密绑在一起。

### 类型与主角边界

思路 3 的作品类型锁定为：

- 港综
- 年代
- 商黑结合
- 智斗高于武斗
- 黑白灰利益交换成立

主角沈轩被明确锁定为：

- 极致利己
- 但不是无脑疯狗
- 讲收益、讲底线、讲代价
- 做事要有算计
- 系统是辅助，不是保姆

也就是说，思路 3 的“planning”并不抽象，它同样携带强烈的内容宪法。

## 用户创作宪法

以下是跨 3 文件抽出的稳定上位法，足够作为 Meta 层候选，且都具有跨作品、跨 skill 复用价值：

1. **不确定就查，不要装懂。**  
2. **任何关键事实必须交叉验证，尤其年代、政策、金融、军事、地理、术语。**  
3. **不能伪造已读文件、已读章节、已读资料。**  
4. **不能机械追满 100 章；上下文读取必须服从因果闭环。**  
5. **把文件系统当长期记忆，把上下文当易失内存。**  
6. **最新正文高于旧设定；当前状态判断优先看当前状态文件。**  
7. **人物行为必须服从“过往经历 + 当前利益 + 性格底色”。**  
8. **禁止为了推剧情让主角、反派、配角突然降智。**  
9. **拒绝机械降神；解决方案必须来自既有铺垫、资源、信息差、规则。**  
10. **Show, don't tell；设定、情绪、强大、压迫都要落到事件、动作、器物、反应。**  
11. **配角不是工具人，必须有算盘、恐惧、筹码、误判与反扑。**  
12. **爽点必须兑现，而且要有因果与代价，不能只喊口号。**  
13. **风格必须去 AI 味：少空心高级词，少模板狠话，少重复句式。**  
14. **长篇必须有账本意识：状态、资源、收益、伏笔、关系都要可追溯。**  
15. **输出不暴露思维链，只给外显自检、依据、结论、结果。**  
16. **联网只补硬事实，不替代作品内已写明的剧情真相。**  
17. **书名、简介、开篇三件套必须协同服务转化，不可各说各话。**  
18. **不同章节类型应用不同压迫方式，不能全书只剩一个战斗模板。**  
19. **女性角色可以重要，但不能被写成后宫挂件或情绪摆设。**  
20. **所有关系变化都要有事件驱动；无理由的爱恨、结盟、背叛都不成立。**

这 20 条里，1-8 更像工作法宪法，9-20 更像创作法宪法。

## 已有 skill 画像

### 已有 skill A：dark-fantasy-ultimate-engine

它不是一个“写玄幻的 prompt”，而是一个带 4 类任务分流、4 套记录系统、1 套收益数值锚点、1 套伏笔池机制、1 套输出表格协议的**长篇玄幻生产 skill**。它的强项是：

- 风格锁定极强
- 当前状态治理成熟
- 资源 / 收益 / 伏笔追踪能力强
- 审稿维度具体
- 很适合高反馈、强因果、黑暗掠夺型爽文

它的增强方向不该是“重做一个更聪明的玄幻写手”，而该是：

- 抽象出可复用 schema
- 抽象出状态文件 contract
- 抽象出自检与章节结算模板
- 给它补更稳的 agent / workflow / 检索支撑

### 已有 skill B：planning-with-files v9.2.0-repo-grounded

它不是一个“教 AI 写计划”的 prompt，而是一个**面向真实仓库的文件化协作 skill**。它的强项是：

- 明确 repo 真相来源
- 明确复杂任务三文件分工
- 明确增量热更新 vs 全量冷启动
- 明确联网边界
- 明确路径纪律与禁伪造要求
- 自带 allowed-tools 与 hooks

它的增强方向也不是“再发明一个计划系统”，而是：

- 把文件工作流标准化为通用平台能力
- 提供更稳定的规划文件模板 / schema / 生命周期
- 把其规则内化为 agent runtime 的默认行为

### 两个已有 skill 的关系

最关键的判断：

- **思路 2 管内容生产真值域**：正文、状态、收益、伏笔、风格。
- **思路 3 管仓库协作真值域**：规划、发现、进度、考据、跨会话接力。

它们并不是平行重复，而是：

- 一个偏**作品内运行时**
- 一个偏**仓库外工作流**

蒸馏时最忌讳把二者揉成一个大而泛的超级 prompt，最后两边都变钝。

## 与原料其他部分的关系

> 说明：本节严格遵守红线，**未扩扫** `_原料/基座/` 与 `_原料/提示词库参考/`。以下判断仅基于 `思路/` 三文件、`notepad.md`、以及 `scout4-pipeline.md` 中已给出的项目内证据做边界推断，不冒充目录实查结果。

### 与基座的关系：更像“上位法 + 调度法”

从项目 notepad 与 scout4 模板可推知，基座更偏大量结构化模板 / agent 能力库；而思路 2/3 提供的是：

- 何时读什么
- 哪个信息源优先
- 冲突如何裁决
- 何时联网
- 什么叫合格正文
- 什么叫合格审稿
- 什么叫合格计划维护

因此，思路目录不是基座模板的替代物，而是其**Meta 约束层与 Platform 调度层**来源。

### 与提示词库参考的关系：更像“筛选器与兼容层”

提示词库参考大概率提供大量场景化 prompt 与阶段流水线；思路 2/3 则定义：

- 哪些场景值得调用
- 调用前要先锁定哪些上下文锚点
- 调用后输出要遵守什么风格 / 事实 / 账本 / 计划纪律

所以，提示词库参考更像“可调用卡片库”，思路目录更像“调用宪法”。

### 思路 2/3 是否已经覆盖其他原料的一部分能力

从文件内容看，**是的，而且覆盖得很深**：

- 思路 2 已经覆盖：正文续写、设定整理、质量审查、账本维护、章节结算、伏笔治理。
- 思路 3 已经覆盖：文件化规划、增量续写、历史考据、资料检索、进度维护、跨会话续接。

这意味着：

- 蒸馏产物里凡是“玄幻正文生产 / 状态维护 / 伏笔池 / 资源账本”相关能力，默认不能替代思路 2，只能兼容增强。
- 凡是“task_plan / findings / progress / repo grounding / 增量热更新 / 不伪造已读”相关能力，默认不能替代思路 3，只能兼容增强。

### 可直接推断的蒸馏边界

#### 必须新增

1. **跨 skill 的统一 Meta 宪法层**：把思路 1/2/3 的共识抽成上位法。  
2. **统一 schema / registry / workflow 描述层**：让 skill 的记录协议、输入契约、输出契约可机器化复用。  
3. **RAG 召回层**：让已有能力有更精准的检索支撑，而不是靠手工点名。  
4. **能力编目与路由层**：让“什么时候调用哪套规则”更稳定。

#### 只能增强

1. `dark-fantasy-ultimate-engine` 的状态卡 / 账本 / 伏笔池机制  
2. `planning-with-files` 的三文件规划体系  
3. 两者的“不要假装读过 / 不要机械追满 / 不要滥用联网 / 不要暴露思维链”纪律

#### 不能替代

1. 用户已经跑通的 skill 名称与核心角色定位  
2. 已有工作流中的真值优先级  
3. 已有仓库型文件落点  
4. 已有作品类型边界与风格锁定  
5. 已有状态账本与伏笔池逻辑

换句话说，蒸馏系统的正确姿势不是“重新发明一个更通用的总 skill”，而是：**把现有 skill 升格为系统中的一级公民。**

## 关键发现与建议

1. **思路目录才是 Meta 层真正来源。**  
   不是基座，也不是提示词库。用户的“什么叫对、什么叫雷、什么叫不能碰”的判断几乎都在这里。

2. **思路 2 和思路 3 已经是两条成熟产品线，不是草稿。**  
   一个解决作品内运行时一致性，一个解决仓库级协作一致性。蒸馏时应保留双 skill 架构。

3. **思路 1 必须被拆成“创作宪法 + 研究纪律 + 避雷字典”。**  
   它太杂，不能原样塞进一个 system prompt；但它又是最关键的母规则池，必须结构化抽取。

4. **“不机械追读 100 章”是跨文件共识，不是偶然表述。**  
   这条应进入系统级硬规则，避免未来蒸馏平台回退成粗暴大上下文策略。

5. **“不伪造已读”是用户极强红线，应上升为 runtime guard。**  
   最好不是只写在文档里，而要体现在 agent 输出模板或检查器中。

6. **思路 2 的账本系统值得单独产品化。**  
   状态卡、微粒账本、伏笔池、章节结算，这套机制已经具备 schema 化价值，完全可以沉淀为 Foundation/Platform 层标准件。

7. **思路 3 的三文件体系值得单独产品化。**  
   `task_plan.md / findings.md / progress.md` 不只是记笔记，而是长篇项目的持久化工作内存，适合升格为标准工作区协议。

8. **两套 skill 的共同硬核不是文风，而是真值治理。**  
   都在强调：谁是当前真相、何时回读、何时扩窗、何时联网、何时更新记录。这部分才是最不可丢的蒸馏资产。

9. **蒸馏时应避免“一锅端”的统一大 prompt。**  
   更推荐四层落点：  
   - Meta：抽创作宪法与研究纪律  
   - Foundation：抽状态 / 伏笔 / 关系 / 计划 schema  
   - Platform：保留两个已有 skill 作为一级能力  
   - RAG：只做召回与补料，不篡改真值优先级

10. **若后续主 agent 要做差异分析，重点不该问“缺什么 prompt”，而该问“哪些规则已经稳定到可以平台化”。**  
    当前最明显可平台化的至少有 3 组：  
   - 状态文件 contract  
   - 规划文件 contract  
   - 外显自检 / 审稿输出 contract

11. **建议后续 Jury 审的不是“写得像不像”，而是“哪些规则必须上升为系统硬边界”。**  
    比如：不伪造已读、最新正文优先、联网只补硬事实、收益必须量化、伏笔必须可追溯。

12. **Scout 2 的最终判断**：  
    `思路/` 目录不是给系统“提供更多灵感”，而是给整个蒸馏工程提供**合法性与边界**。如果这部分蒸馏错了，后面基座再强、RAG 再大，产物也会跑偏。
