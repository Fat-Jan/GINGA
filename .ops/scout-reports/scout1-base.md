# Scout 1: 基座目录深扫报告

结论先行：`_原料/基座/` 不是一堆平级 prompt，而是一个已经半结构化的**小说能力库**。它内部至少分成三层：
1. **题材 profile 层**：`题材/网文/*.md` 提供题材级读者期待、黄金三章、留存/爽点/禁忌参数，像“可配置 profile”。
2. **通用模板层**：`模板/` 提供跨题材可复用的数据骨架，如角色卡、主大纲、章纲、世界观、伏笔、悬念、创意简报。
3. **方法论/校验器层**：`方法论/` 提供市场趋势、平台规范、母题目录、命名策略、反 AI 味、状态 schema 等“指导与约束”，本质上更接近 rulebook / checker / registry 说明，而非正文生成 prompt。

这说明主 agent 在阶段 2 不应该把“基座”直接并入单一 RAG，而应拆成：**profile + template + methodology/checker** 三类资产，分别落到 Foundation 与 Platform 层。

## Schema 草案

我的建议不是给所有基座文件套一张死板 frontmatter，而是定义一张**统一主 schema + 按资产类型扩展字段**。原因：题材 profile、模板骨架、方法论文档的职责不同，但它们又确实共享部分检索/编排字段。

### 推荐统一 YAML frontmatter

| 字段 | 类型 | 必填 | 取值范围 | 来源 |
|---|---|---:|---|---|
| `id` | string | 是 | 唯一 slug，建议 `base-xxx` | 新增治理字段 |
| `title` | string | 是 | 文档标题 | `# 标题` |
| `asset_type` | enum | 是 | `genre_profile` / `template` / `methodology` / `checker` / `schema_ref` | 由目录 + 内容判断 |
| `source_path` | string | 是 | 仓内相对路径 | 文件路径 |
| `topic` | string/null | 否 | 如 `规则怪谈`、`年代`、`都市异能` | 题材文件名 / 标题 |
| `topic_bucket` | enum | 否 | `网文` / `通用` / `方法论` | 一级目录 |
| `stage` | string/null | 否 | `创意阶段` / `设定阶段` / `框架阶段` / `创作阶段` / `进阶技巧` / `数据分析` / `商业化` / `辅助工具` / `宏观` / `中观` / `微观` / `势力` / `辅助` / `设定` / `项目` | 文件名或正文分层 |
| `use_case` | string/null | 否 | 如 `角色卡` / `主大纲` / `读者反馈` / `黄金三章` / `世界观构建` | 文件名 / H2 |
| `abstraction_level` | enum | 是 | `macro` / `meso` / `micro` / `meta` / `runtime` | 文件名、内容定位 |
| `platform_scope` | string[] | 否 | `番茄` / `七猫` / `起点` / `晋江` / `知乎` / `红袖` / `豆瓣阅读`... | 方法论文本 |
| `input_contract` | string[] | 否 | 上游需要准备什么 | 显式依赖 / 使用说明 |
| `output_contract` | string[] | 否 | 产出哪些结构 | 模板字段 / 输出要求 |
| `state_entities` | string[] | 否 | `characters` / `world` / `outline` / `foreshadow` / `hook` / `platform_config`... | `state-schema`、模板骨架 |
| `dependencies` | string[] | 否 | 依赖的其它模板/资料路径 | 显式引用 |
| `quality_checks` | string[] | 否 | 必查项 / 禁忌 / 评分点 | 质量清单 / 禁忌清单 |
| `hook_types` | string[] | 否 | `危机钩` / `悬念钩` / `情绪钩` 等 | 题材 profile / 吸引力分类法 |
| `cool_point_types` | string[] | 否 | 爽点类型列表 | 题材 profile |
| `reader_profile` | object/null | 否 | 年龄、性别、阅读偏好、付费触发点 | 题材 profile |
| `reuse_scope` | enum | 是 | `cross_genre` / `genre_specific` / `platform_specific` / `era_specific` | 扫样归类 |
| `safety_level` | enum | 否 | `normal` / `sensitive` / `redline-heavy` | 内容敏感性判断 |
| `status` | enum | 否 | `draft` / `active` / `deprecated` | 后续治理字段 |
| `notes` | string[] | 否 | 迁移备注 | 后续治理字段 |

### 按资产类型的扩展字段

#### 1. `genre_profile`
适用于 `题材/网文/*.md`。

建议追加：
- `golden_three_outline`
- `retention_targets`
- `emotion_curve`
- `chapter_structure`
- `hook_strategy`
- `taboos`
- `style_guide`
- `opening_template_available`（如部分文件带“附录：开篇模板库”）

#### 2. `template`
适用于 `模板/*/*.md`。

建议追加：
- `template_family`：`character` / `outline` / `foreshadow` / `suspense` / `worldbuilding` / `project`
- `fields_required`
- `fields_optional`
- `supports_incremental_update`（如主角卡、伏笔、悬念都带历史/动态更新）
- `render_format`：`yaml` / `markdown_table` / `mermaid` / `xml`

#### 3. `methodology` / `checker` / `schema_ref`
适用于 `方法论/`。

建议追加：
- `method_family`：`writing` / `market` / `platform` / `creativity`
- `normative_strength`：`guide` / `hard_rule` / `schema`
- `targets`：面向哪些题材/平台/阶段
- `integrates_with`：如 `creative-brief`、`state-schema`、`golden-three-checker`

### 核心判断

这套 schema 的关键不在“补 metadata”，而在把基座拆成**可路由资产**：
- `genre_profile` 给 Meta/Platform 作为约束参数；
- `template` 给 Foundation/Platform 作为结构骨架；
- `methodology/checker` 给 Platform 作为前置校验和质量门。

8. `support`：灵感捕捉/卡文急救/实体图/辅助生成

### 5. 重要分布特征

1. **题材 profile 结构高度统一**：25 个样本几乎都共享同一 H2 骨架，这说明它们可以被标准化抽字段，而不是只做全文检索。
2. **模板目录是“实体/结构表单”而非 prompt 文本**：尤其角色、世界观、伏笔、悬念，本质更接近 schema 表。
3. **方法论目录内部异质性最大**：有指南、有 checker、有状态 schema、有平台规范，后续必须细分 asset_type，不然会把规则文误当写作模板。

## 思维链层级模式

严格说，当前我抽读的很多“基座”文件并不都显式写了 `## 系统角色 / ## 思维链指令 / ## 输出要求` 这组三段式；那种格式更多见于编号型题材簇和某些专用模板。**但从内容组织看，它们仍然呈现出稳定的隐式思维链层级**。

### 模式 A：题材 profile 的 7 层链路

在 `题材/网文/*.md` 里，最稳定的顺序是：

1. **题材价值判断**：核心爽点、读者期待、禁雷
2. **开篇设计**：黄金三章
3. **推进约束**：阻力、代价、残留风险
4. **体验参数化**：读者画像、留存漏斗、情绪弧线、钩子系统、章节结构、平台策略
5. **局部激励设计**：爽点模式与密度
6. **质量门**：质量清单、必须满足、绝对不能
7. **语言风格**：对话风格、描写重点、语料示例

这是一种从“读者预期 → 结构 → 参数 → 风格”的上到下链条，适合转成 profile config。

### 模式 B：模板骨架的 4-6 层链路

#### 角色卡类
以 `主角.md`、`反派.md`、`角色卡通用模板.md` 为代表：

1. 基本身份
2. 性格/说话风格
3. 背景与动机
4. 能力与限制
5. 关系网络
6. 动态状态/成长轨迹

这说明角色模板天然支持**静态设定 + 运行时状态**双层模型，不是一次性设定卡。

#### 大纲类
以 `主大纲.md`、`章节大纲.md` 为代表：

1. 基础元信息
2. 结构骨架
3. 节点/场景拆解
4. 伏笔/设定/状态联动
5. 节奏校验
6. 快速模板输出

特别是 `章节大纲.md` 已经内置“本章阻力 / 代价 / 局部收益 / 残留风险”，说明它不是普通章纲，而是**带冲突经济学**的章级计划器。

#### 世界观/项目类
以 `世界观构建.md`、`创意简报.md` 为代表：

1. 核心 premise
2. 设定骨架
3. 关系/母题/机制
4. 平台与卖点定位
5. 示例/流程/自动生成接口

`创意简报.md` 比普通模板更像一个**初始化协议**：它把母题选择、平台定位、题材卖点、古代补充真值等前置到开书入口。

### 模式 C：方法论文档的 3 类链路

#### 1. 规则型
如 `古代官职体系.md`、`古代家族亲缘.md`：
- 目的
- 最小原则/真值表
- 常见错误组合
- 生成前检查
- 一票否决项

#### 2. 指南型
如 `平台分析指南.md`、`投稿指南.md`：
- 平台画像
- 数据/审核/风格要求
- 平台差异对比
- 输出模板或自查清单

#### 3. schema/checker 型
如 `state-schema.md`、`读者吸引力分类法.md`：
- 概念定义
- 字段结构
- 评分/约束矩阵
- 与运行时状态或检查器对接

### 可抽象的统一 CoT 层级

如果后续要把“思维链指令”标准化，我建议统一抽成：

1. **Context**：题材/平台/用户目标
2. **Premise/Truth**：世界、角色、核心设定真值
3. **Constraint**：禁忌、审校点、平台红线
4. **Planning**：黄金三章/卷纲/章纲/节奏/钩子
5. **Execution**：正文、场景、对话、描写
6. **Review**：质量清单、留存、吸引力、反 AI 味
7. **State Update**：角色状态、伏笔状态、关系图、summary/state schema

这 7 层比原始目录名更稳，更适合抽成 agent workflow DSL。

## 模板依赖关系示例

我只写有内容依据的依赖，不写空泛“可能相关”。

### 依赖示例 1：`模板/项目/创意简报.md` → `方法论/写作/世界观母题目录.md`

显式证据：`创意简报.md` 在“母题选择”段直接引用 `shared/references/writing/worldview-motif-catalog.md`，要求先选世界观母题/机制母题/包装关系母题，再继续立项。

结论：`创意简报` 不是独立模板，而是**以母题目录作为外部枚举表**的初始化入口。

### 依赖示例 2：`方法论/写作/世界观母题目录.md` → `模板/项目/创意简报.md`

显式证据：该文档用途直接写给 `creative-brief`、`novel-init`、`novel-outline`、`novel-package`，并明确“再回填 `shared/templates/project/creative-brief.md`”。

结论：二者形成双向依赖：母题目录提供候选空间，创意简报负责实例化选择。

### 依赖示例 3：`模板/大纲/主大纲.md` → 家族/官制真值表

显式证据：主大纲中明确写了：如果是宫斗宅斗/古代言情/历史家族权力路线，要先完成 `设定集/家族/宅门真值表.md`、`设定集/家族/小型家谱.md`；若涉及朝堂官场，再完成 `设定集/官制/官职真值表.md`、`设定集/官制/权力层级图.md`。

结论：主大纲依赖外部真值表先锁设定，说明它不是“从零脑补的大纲模板”，而是**真值约束下的结构编排器**。

### 依赖示例 4：`方法论/创意/黄金三章.md` → checker / precheck / platform config

显式证据：文末直接挂出：
- `references/creativity/platform-golden-three-check-table.md`
- `bees/golden-three-checker.md`
- `commands/novel-precheck.md`
- `config/platform-config.md`

结论：这不是纯理论文，而是**明确预期接入检查器和平台配置**的规则核心。

### 依赖示例 5：`方法论/市场/读者吸引力分类法.md` → `方法论/市场/state-schema.md`

显式证据：文中多次写“完整结构定义参考 `state-schema.md`”、“详见 `state-schema.md` 第四章”。

结论：吸引力分类法并不打算独立存在，而是要挂到运行时 state 里成为评分字段或 checker 输入。

### 依赖示例 6：`方法论/市场/state-schema.md` → 平台配置 / 组合定位 sidecar / 实体模板

显式证据：`state-schema.md` 明写 `platform_config` 负责平台级硬边界，另有 `content-positioning` sidecar，并专门定义 `entities.characters.protagonist` 等结构。

结论：它是把“题材 profile + 平台规则 + 角色世界实体”汇总成统一运行时状态的核心桥梁。

### 依赖示例 7：`题材/网文/*.md` → `方法论/创意/黄金三章.md` / 平台分析文档（隐式强依赖）

证据方式：25 个题材 profile 都稳定内嵌“黄金三章 / 留存 / 钩子 / 平台策略 / 质量清单”这些结构，且字段命名高度一致。

结论：虽然未逐个显式链接，但它们显然是把“黄金三章方法论 + 平台分析 + 吸引力参数”实例化到具体题材后的结果，可视为**profile 化下游产物**。

### 依赖示例 8：角色模板 / 伏笔模板 / 悬念模板 / 实体关系图 → `state-schema.md`

证据方式：`主角.md` 有“当前状态（动态更新）”、“状态历史”；`伏笔模板.md`、`悬念模板.md` 有状态历史；`实体关系图.md` 有实体索引和关系索引；`state-schema.md` 则定义 characters/items/locations/factions 等统一状态结构。

结论：这些模板是 state-schema 的**人工填写前端**，state-schema 是它们的**机器聚合后端**。

### 依赖模式归纳

整体上，基座里的依赖不是树，而是三角结构：
- **方法论** 给枚举、规则、评分、平台边界
- **模板** 给结构化填空骨架
- **题材 profile** 给参数化题材配置

真正生成时，应该是三者合流，而不是单看其中一层。

## 方法论跨题材复用度

这里我只评估 `方法论/` 目录，不把模板和题材 profile 混进来。

### 总体判断

在本次实际阅读的 13 个方法论文件里：

- **高复用（跨大多数题材）**：8 个，约 **62%**
- **中复用（跨多个题材，但依赖平台/时代/风格）**：3 个，约 **23%**
- **低复用（明显绑定特定背景）**：2 个，约 **15%**

### 分层判断表

| 文件 | 复用度 | 原因 |
|---|---|---|
| 世界观母题目录 | 高 | 母题本身跨题材 |
| 反AI味 | 高 | 几乎所有正文生成都可套用 |
| 角色命名 | 高 | 除极小部分特殊风格，通用性强 |
| 平台分析指南 | 高 | 只要要发平台就适用 |
| 投稿指南 | 高 | 平台投稿硬约束，跨题材通用 |
| 黄金三章 | 高 | 开篇结构普适，但平台/男女频有差异 |
| 2026网文市场趋势 | 高 | 题材选择与卖点判断都能复用 |
| 读者吸引力分类法 | 高 | 钩子/爽点/追读力是跨题材通用语义 |
| state-schema | 中高 | 运行时结构通用，但落地要按题材裁剪字段 |
| 跨平台写作规范 | 中 | 跨平台通用，但更偏发表场景 |
| 题材跨平台适配矩阵 | 中 | 对题材选择极有用，但不是每步都调用 |
| 古代官职体系 | 低 | 主要服务古代/宫斗/历史线 |
| 古代家族亲缘 | 低 | 主要服务古代家族权力题材 |

### 关键发现

1. **方法论目录不是“都能直接跨题材复用”**。
   - 里面至少有一批是通用规则（黄金三章、反 AI 味、吸引力分类法）。
   - 另一批是专题校验器（古代官职、家族亲缘），应按条件路由。

2. **最值得抽成系统能力的不是长文本本身，而是其中的结构化约束**。
   - 例如 `黄金三章` 可抽成开篇 checker。
   - `读者吸引力分类法` 可抽成评分器。
   - `角色命名` 可抽成命名器。
   - `古代官职体系` 可抽成古代题材校验门。

3. **方法论的正确角色应是“调用条件受控的 rule modules”**，不是在每次生成时整篇注入上下文。

## 关键发现与建议

### 关键发现

1. **基座内部已经天然分层**。
   - `题材/网文` 是 profile 参数层。
   - `模板/` 是实体/结构骨架层。
   - `方法论/` 是规则/校验/schema 层。
   - 这三层已经非常接近主 agent 目标中的 Foundation + Platform 边界。

2. **题材 profile 高度标准化，可以批量抽字段。**
   几乎所有已读题材文档都稳定包含：
   - 题材概述
   - 黄金三章
   - 推进阻力与代价
   - 读者体验配置
   - 爽点模式
   - 禁忌清单
   - 写作风格

   这意味着题材 profile 很适合变成 `profile.yaml + longform notes` 的双轨资产，而不是保留纯 markdown 散文。

3. **模板目录最像 Foundation schema。**
   特别是角色、世界观、伏笔、悬念、大纲、关系图，已经接近数据库表单/状态真值格式。

4. **方法论目录要强拆子类型。**
   不能简单归成“方法论”。至少要拆：
   - guide
   - checker
   - schema_ref
   - platform_rule
   - domain_specific_rule

5. **state-schema 是极关键桥梁。**
   它证明原作者已经不满足于 prompt 堆砌，而在往 runtime state 方向走。阶段 2/4 应该优先吸收这条线。

6. **基座已有大量“隐式工作流接口”。**
   例如 `创意简报 -> 母题目录 -> 主大纲 -> 章节大纲 -> 角色/世界/伏笔/悬念 -> state-schema/checker`。这条链路非常适合主 agent 后续画成 DAG。

### 风险与坑

1. **文件命名体系不统一。**
   目录里既有 slug 英文题材 profile，也有中文编号型簇，也有纯模板名、纯方法论文。后续必须用 frontmatter 统一，不要继续依赖文件名解析。

2. **方法论文本太长，直接喂模型会污染上下文。**
   特别是 `反AI味.md`、`state-schema.md`、`读者吸引力分类法.md`。应拆成 rules/checks/schema sections 做按需检索。

3. **题材 profile 与方法论存在重复表达。**
   黄金三章、平台策略、爽点/钩子定义在上游方法论和下游题材 profile 都出现。后续要明确“上游真源 vs 下游实例化”。

4. **部分依赖指向仓内不存在于本目录的路径。**
   如 `shared/templates/...`、`shared/references/...`、`bees/...`、`commands/...`、`config/...`。蒸馏时必须把这类引用映射成逻辑依赖，而不是死认文件路径。

### 给主 agent 的建议

#### 建议 1：先把基座拆成 4 类资产注册表

建议 registry 至少有：
- `genre_profiles`
- `templates`
- `method_rules`
- `checkers_and_schema_refs`

#### 建议 2：阶段 2 优先吸收这几条主线

1. `创意简报 + 世界观母题目录`
2. `主大纲 + 章节大纲`
3. `角色/世界观/伏笔/悬念/关系图`
4. `黄金三章 + 读者吸引力分类法 + state-schema`
5. `平台规则 + 投稿规范 + 命名策略`

#### 建议 3：把题材 profile 变成参数文件，不保留纯文本唯一形态

最理想的落地方式是：
- `profile.yaml`：留结构字段（留存、爽点密度、钩子类型、禁忌、平台策略）
- `guide.md`：保留长文本示例、语气、风格说明

#### 建议 4：把方法论拆成“调用条件”

例如：
- 命中古代/宫斗/历史家族 → 加载 `古代官职体系` + `古代家族亲缘`
- 命中投稿/平台发布 → 加载 `投稿指南` + `跨平台写作规范`
- 命中开篇策划 → 加载 `黄金三章` + 对应题材 profile
- 命中检查/复盘 → 加载 `读者吸引力分类法` + `state-schema`

#### 建议 5：不要把“基座=prompt 仓库”当结论

更准确的定义应是：

> 基座 = 一套以题材 profile、结构模板、方法论规则、状态 schema 为核心的小说创作能力资产库。

这个判断对后续四层架构非常关键：
- Meta：吸收方法论中的红线/平台/审美约束
- Foundation：吸收模板和 state schema
- Platform：吸收依赖关系、checker、workflow
- RAG：只作为这三类资产的召回辅助，不做真值源
